# 一覧カラム削減 画面別仕様書 2026-06-04

## 1. 目的

UAT画像レビューで、一覧系UIのカラム数が多く、利用者が対象を探す・選ぶ前に内部情報を読み解く必要があることが判明した。本仕様は、一覧を「探す・選ぶ」に特化させ、選択後に詳細ペイン、選択カード、tooltipで全データを確認できる構成へ整理する。

本PRは docs-only とし、既存コードは変更しない。

## 2. 共通方針

- 一覧は「探す・選ぶ」に必要な業務項目、状態、件数だけを表示する。
- 公開ID、内部ID、備考、検索用表記、長い日時は一覧から外す。
- 一覧から外した情報は、行選択後の詳細ペイン、選択カード、またはtooltipで確認できるようにする。
- 詳細ペインには、対象レコードの全データを欠落なく表示する。
- IDは通常操作の判断材料ではなく、問い合わせ、監査、障害調査、データ照合時に確認する補助情報として扱う。
- 日時表示は将来的に `YYYY/MM/DD HH:MM:SS` に統一する。実装時は一覧・詳細・tooltip・監査ログで同じフォーマットを使う。
- カラム削減後も検索条件は劣化させない。検索用表記、備考、公開ID、内部IDで検索できる場合は、表示しなくても検索対象として維持する。
- viewer/editor/admin のRBAC表示制御は変更しない。

## 3. 対象画面

### 3.1 検索

#### 現状の問題

- 検索結果一覧に内部ID、公開ID、検索用表記、更新日時、備考が並び、名前を探す主目的より補助情報が目立つ。
- 関連一覧にも内部リンクID、リンク公開IDが表示され、タイトル・管理番号・サブタイトルの把握を妨げる。
- 備考や日時が長い場合、一覧幅を圧迫して重要な名前・関連情報が読みづらくなる。

#### 一覧に残すカラム

検索結果一覧:

| カラム | 理由 |
|---|---|
| 名前 | 検索対象の主識別子 |
| 状態 | 有効/削除済みなど、選択可否判断に必要 |
| タイトル関連数 | 関連有無の把握に必要 |
| サブタイトル関連数 | 関連有無の把握に必要 |
| 関連合計 | 絞り込み・優先確認の判断に必要 |

選択中名前の関連一覧:

| カラム | 理由 |
|---|---|
| タイトル | 関連先の主識別子 |
| 管理番号 | サブタイトルを区別する業務項目 |
| サブタイトル | 関連先の主識別子 |

#### 一覧から外すカラム

- 内部ID
- 公開ID
- 検索用表記
- 更新日時
- 備考
- 内部リンクID
- リンク公開ID

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中名前カード: 内部ID、公開ID、名前、検索用表記、状態、作成日時、更新日時、備考、タイトル関連数、サブタイトル関連数、関連合計。
- 関連行tooltip: 内部リンクID、リンク公開ID、タイトル公開ID、サブタイトル公開ID、関連作成日時、関連更新日時。
- 詳細ペイン: 選択中名前の全項目と、関連タイトル/サブタイトルの全項目を確認できる構成にする。

#### 受け入れ条件

- 検索結果一覧は、初見で名前と関連件数を横方向スクロールなしに確認できる。
- 内部ID、公開ID、検索用表記、備考、長い日時が一覧の主要カラムとして表示されない。
- 検索結果を1行選択すると、外した全情報を詳細ペインまたは選択カードで確認できる。
- 関連一覧はタイトル、管理番号、サブタイトルの確認に集中できる。
- 公開IDや備考を検索条件として使える既存挙動がある場合、表示削減後も検索できる。

#### テスト観点

- 名前、検索用表記、備考、公開IDで検索できること。
- 検索結果の一覧ヘッダーに削除対象カラムが残っていないこと。
- 行選択時に詳細ペインへ内部ID、公開ID、検索用表記、備考、日時が表示されること。
- 関連行のtooltipまたは詳細表示でリンクID系情報を確認できること。
- 日時が表示される箇所は `YYYY/MM/DD HH:MM:SS` へ統一可能な表示経路になっていること。

### 3.2 名前管理

#### 現状の問題

- 名前一覧に内部ID、公開ID、検索用表記、備考が並び、名前と状態を選ぶ操作に対して情報量が過多である。
- 検索用表記と備考は検索・確認には必要だが、常時一覧に出すと列幅を圧迫する。
- 関連件数が複数列で並ぶため、名前そのものの視認性が下がる。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| 名前 | 管理対象の主識別子 |
| 状態 | 更新/削除判断に必要 |
| タイトル関連数 | 関連有無の判断に必要 |
| サブタイトル関連数 | 関連有無の判断に必要 |
| 関連合計 | 対象選択の優先度判断に必要 |

#### 一覧から外すカラム

- 内部ID
- 公開ID
- 検索用表記
- 備考

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中名前カード: 内部ID、公開ID、名前、検索用表記、状態、備考、作成日時、更新日時、削除日時、関連件数。
- 編集フォーム近傍: 検索用表記、備考は選択後に確認・編集できるようにする。
- 関連件数tooltip: タイトル関連数、サブタイトル関連数、関連合計の意味を短く補足する。

#### 受け入れ条件

- 名前一覧は名前、状態、関連件数だけで対象を探せる。
- 公開ID、内部ID、検索用表記、備考が一覧から外れている。
- 行選択後、外した全情報を選択カードまたは編集フォームで確認できる。
- 検索入力の対象には検索用表記と備考を維持する。

#### テスト観点

- 一覧ヘッダーが残すカラムだけで構成されていること。
- 名前を選択すると、検索用表記と備考がフォームまたは詳細に反映されること。
- 公開ID/内部IDが詳細表示で確認できること。
- 関連件数の表示が既存データ件数と一致すること。
- viewer/editor/admin で許可済み操作の可否が変わらないこと。

### 3.3 タイトル管理

#### 現状の問題

- タイトル一覧に内部ID、公開ID、更新日時、備考、関連する名前が並び、タイトル名と状態を選ぶ目的に対して列数が多い。
- 関連する名前が長い場合、タイトル名や状態の表示領域を奪う。
- 更新日時は監査・確認用には必要だが、対象選択時の主要判断材料ではない。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| タイトル名 | 管理対象の主識別子 |
| 状態 | 更新/削除/サブタイトル作成可否の判断に必要 |
| 関連名前数 | 関連有無を簡潔に示す |
| サブタイトル数 | 配下データの有無を簡潔に示す |

`関連名前数` と `サブタイトル数` は、実装時に取得できる範囲で追加または既存集計を流用する。集計が未整備の場合は、タイトル名と状態を最小構成とし、関連する名前は選択後表示に移す。

#### 一覧から外すカラム

- 内部ID
- 公開ID
- 更新日時
- 備考
- 関連する名前の全文

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中タイトルカード: 内部ID、公開ID、タイトル名、状態、備考、作成日時、更新日時、削除日時、関連する名前一覧、サブタイトル件数。
- 関連名前tooltipまたは詳細ペイン: 関連する名前の一覧、各名前の公開ID、状態。
- 編集フォーム: 備考は選択後に確認・編集できるようにする。

#### 受け入れ条件

- タイトル一覧はタイトル名と状態が最も読みやすい構成になっている。
- 関連する名前の長い文字列が一覧を横に広げない。
- 公開ID、内部ID、備考、長い日時が一覧から外れている。
- 行選択後、関連する名前とID系情報を詳細で確認できる。

#### テスト観点

- タイトル一覧ヘッダーに削除対象カラムが残っていないこと。
- タイトル選択時に関連する名前一覧が詳細に表示されること。
- 削除済みタイトル選択時のサブタイトル操作制御が維持されること。
- 備考と日時が詳細で確認できること。
- 関連名前数/サブタイトル数を表示する場合、件数が実データと一致すること。

### 3.4 サブタイトル管理

#### 現状の問題

- サブタイトル一覧に内部ID、公開ID、タイトル内部ID、更新日時、備考が並び、親タイトル、管理番号、サブタイトル名を選ぶ操作が埋もれる。
- 親タイトルとタイトル内部IDの両方があるため、業務上必要な親タイトル名より内部情報が目立つ。
- 表示順は一覧で必要だが、更新日時・備考と同列に並ぶと確認対象が散る。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| 親タイトル | 所属先の識別に必要 |
| 管理番号 | サブタイトルの業務識別子 |
| サブタイトル名 | 管理対象の主識別子 |
| 状態 | 更新/削除判断に必要 |
| 表示順 | 並び順確認に必要 |

#### 一覧から外すカラム

- 内部ID
- 公開ID
- タイトル内部ID
- 更新日時
- 備考

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中サブタイトルカード: 内部ID、公開ID、タイトル内部ID、タイトル公開ID、親タイトル、管理番号、サブタイトル名、状態、表示順、備考、作成日時、更新日時、削除日時。
- 親タイトルtooltip: タイトル公開ID、タイトル状態、タイトル備考。
- 編集フォーム: 管理番号、サブタイトル名、表示順、備考を選択後に確認・編集できるようにする。

#### 受け入れ条件

- サブタイトル一覧は親タイトル、管理番号、サブタイトル名、状態、表示順に集中している。
- 公開ID、内部ID、タイトル内部ID、備考、長い日時が一覧から外れている。
- サブタイトル選択後、親タイトルを含む全情報を詳細で確認できる。
- 親タイトル選択とサブタイトル選択の現在状態が明確に分かる。

#### テスト観点

- 一覧ヘッダーが指定カラムに整理されていること。
- 親タイトル、管理番号、サブタイトル名、備考で検索できること。
- サブタイトル選択時に内部ID、公開ID、タイトル内部ID、備考、日時が詳細に表示されること。
- 表示順の値が一覧と編集フォームで一致すること。
- 削除済み親タイトル選択時の操作制御が維持されること。

### 3.5 関連付け

#### 現状の問題

- 関連付け画面は登録/解除対象の選択が主目的だが、候補表示やtooltipにID情報が混在すると、名前、タイトル、サブタイトルの関係把握が難しくなる。
- 候補が長い場合、公開IDや内部IDが見えていると選択肢の主要テキストが読みにくい。
- 登録対象と解除対象で、確認すべき情報の粒度が揃っていない。

#### 一覧に残すカラム

関連付け候補または選択肢:

| 表示項目 | 理由 |
|---|---|
| 名前 | 関連元の主識別子 |
| タイトル | 関連先の主識別子 |
| 管理番号 | サブタイトル識別に必要 |
| サブタイトル | 関連先の主識別子 |
| 状態 | 関連付け可否判断に必要 |

解除対象一覧または選択肢:

| 表示項目 | 理由 |
|---|---|
| 名前 | 解除対象の主識別子 |
| タイトル | 解除対象の主識別子 |
| 管理番号 | サブタイトル識別に必要 |
| サブタイトル | 解除対象の主識別子 |

#### 一覧から外すカラム

- 内部リンクID
- リンク公開ID
- 名前内部ID
- 名前公開ID
- タイトル内部ID
- タイトル公開ID
- サブタイトル内部ID
- サブタイトル公開ID
- 関連作成日時
- 関連更新日時
- 備考

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 登録対象カード: 選択中名前、選択中タイトル/サブタイトル、状態、関連可否、各公開ID、各内部ID。
- 解除対象カード: リンクID、リンク公開ID、名前、タイトル、管理番号、サブタイトル、関連作成日時、関連更新日時。
- tooltip: 長い名前/タイトル/サブタイトルの全文、公開ID、内部IDを確認できる補助表示。

#### 受け入れ条件

- 候補選択時の主表示は名前、タイトル、管理番号、サブタイトルに集中している。
- ID系情報と日時が候補テキストの先頭や一覧カラムとして表示されない。
- 登録前/解除前に、選択カードで対象の全情報を確認できる。
- 登録不可・解除不可の理由が状態またはメッセージで分かる。

#### テスト観点

- 登録候補と解除候補の表示文字列に不要なIDが混在しないこと。
- 候補選択後、選択カードまたはtooltipで各内部ID/公開IDを確認できること。
- 登録/解除の実行前確認に名前、タイトル、管理番号、サブタイトルが含まれること。
- viewerでは登録/解除フォームが表示または有効化されない既存制御が維持されること。
- 関連付け後の候補再読込で表示項目が更新されること。

### 3.6 削除データ

#### 現状の問題

- 削除データ一覧に内部ID、公開ID、削除日時、詳細が並び、復元/完全削除対象を選ぶ操作に対して情報量が多い。
- 詳細列は内容が長くなりやすく、分類、表示名、タイトル、管理番号の視認性を下げる。
- 削除日時は確認に必要だが、長い日時が一覧幅を圧迫する。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| 分類 | 名前/タイトル/サブタイトル/リンクの識別に必要 |
| 表示名 | 復元/完全削除対象の主識別子 |
| タイトル | タイトル配下データやリンクの識別に必要 |
| 管理番号 | サブタイトル識別に必要 |

削除日時を一覧に残す場合は短縮表示または相対表示にし、正式日時は詳細に表示する。将来の統一後は `YYYY/MM/DD HH:MM:SS` を詳細で必ず表示する。

#### 一覧から外すカラム

- 内部ID
- 公開ID
- 詳細
- 長い削除日時

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中削除データカード: 内部ID、公開ID、分類、表示名、タイトル、管理番号、詳細、削除日時、元データの備考、検索用表記。
- 復元/完全削除確認ダイアログ: 分類、表示名、タイトル、管理番号、削除日時、公開IDを表示する。
- tooltip: 詳細全文、公開ID、内部ID。

#### 受け入れ条件

- 復元/完全削除対象を分類、表示名、タイトル、管理番号で探せる。
- 内部ID、公開ID、詳細全文、長い日時が一覧を圧迫しない。
- 対象選択後、削除日時と詳細全文を確認できる。
- 復元/完全削除前に誤操作防止に必要な対象情報が表示される。

#### テスト観点

- 削除データ一覧ヘッダーに内部ID、公開ID、詳細が残っていないこと。
- 名前、タイトル、サブタイトル、リンクの各分類で選択カードに全情報が表示されること。
- 復元/完全削除確認で対象が一意に分かること。
- 削除日時が詳細で `YYYY/MM/DD HH:MM:SS` 形式に統一可能な表示になっていること。
- admin以外の権限制御が変わらないこと。

### 3.7 監査ログ

#### 現状の問題

- 監査ログ一覧に内部ID、公開ID、内部対象ID、長い実行日時があり、操作履歴を追ううえで重要なデータ種類と操作が埋もれる。
- ID系情報は調査時に必要だが、一覧で常時表示すると時系列確認と絞り込み結果確認の速度が落ちる。
- 変更前/変更後JSONが詳細領域にあるため、一覧は操作概要に特化できる。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| データ種類 | 対象種別の把握に必要 |
| 実行した操作 | 操作内容の把握に必要 |
| 実行日時 | 時系列確認に必要 |

実行日時は将来的に `YYYY/MM/DD HH:MM:SS` に統一する。幅が問題になる場合は一覧側を同形式のまま適切な列幅にし、詳細にも同じ日時を表示する。

#### 一覧から外すカラム

- 内部ID
- 公開ID
- 内部対象ID

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中操作カード: 内部ID、公開ID、データ種類、内部対象ID、実行した操作、実行日時。
- 詳細ペイン: 変更前JSON、変更後JSON、差分確認に必要な全フィールド。
- tooltip: 公開ID、内部対象ID、実行日時。

#### 受け入れ条件

- 監査ログ一覧はデータ種類、操作、実行日時だけで時系列確認できる。
- ID系情報が一覧から外れている。
- 行選択後、ID系情報と変更前/変更後JSONを詳細で確認できる。
- 絞り込み条件は既存のデータ種類、操作、日時条件を維持する。

#### テスト観点

- 監査ログ一覧ヘッダーに内部ID、公開ID、内部対象IDが残っていないこと。
- ログ選択時に詳細カードへID系情報が表示されること。
- before/after JSON表示が削減後も欠落しないこと。
- 日時フィルタと表示日時が整合すること。
- admin専用表示のRBAC制御が維持されること。

### 3.8 ユーザー管理

#### 現状の問題

- ユーザー一覧に操作者ID、表示名、権限、認証方式、Windowsアカウント、状態、最終ログイン、失敗回数、公開ID、Windows SIDが並び、管理対象の選択に対して列数が多い。
- 公開IDとWindows SIDは長く、一覧幅を大きく消費する。
- 最終ログインの長い日時は確認用には必要だが、常時一覧に出すと主要項目が見づらい。

#### 一覧に残すカラム

| カラム | 理由 |
|---|---|
| 操作者ID | ユーザーの主識別子 |
| 表示名 | 人が確認する補助識別子 |
| 権限 | 管理判断に必要 |
| 認証方式 | Windows/localの識別に必要 |
| 状態 | 有効/無効の判断に必要 |
| 失敗回数 | ロック/認証問題の一次判断に必要 |

Windowsアカウントは、運用で操作者IDと異なる重要情報として必要な場合のみ一覧に残す。残す場合でも公開IDやSIDより優先し、長い値はtooltipで全文確認する。

#### 一覧から外すカラム

- 公開ID
- Windows SID
- 長い最終ログイン日時
- Windowsアカウントの全文表示

#### 詳細ペイン/選択カード/tooltipへ移す情報

- 選択中ユーザーカード: 操作者ID、表示名、権限、認証方式、Windowsアカウント、状態、最終ログイン、失敗回数、公開ID、Windows SID、作成日時、更新日時。
- tooltip: Windowsアカウント全文、Windows SID、公開ID。
- 詳細ペイン: adminがユーザー更新前に全項目を確認できる構成にする。

#### 受け入れ条件

- ユーザー一覧は操作者ID、表示名、権限、認証方式、状態を中心に対象選択できる。
- 公開IDとWindows SIDが一覧を横に広げない。
- 行選択後、公開ID、Windows SID、最終ログインを確認できる。
- admin専用画面であること、viewer/editorでは表示されないことが維持される。

#### テスト観点

- ユーザー一覧ヘッダーに公開ID、Windows SID、長い最終ログイン日時が残っていないこと。
- ユーザー選択時に公開ID、Windows SID、最終ログインが詳細表示されること。
- Windows/local認証ユーザーの表示差異が正しく確認できること。
- 失敗回数、状態、権限の表示が実データと一致すること。
- viewer/editor/admin のタブ表示制御が変わらないこと。

## 4. 共通受け入れ条件

- 対象8画面で、一覧は「探す・選ぶ」に必要な情報へ削減されている。
- 公開ID、内部ID、備考、検索用表記、長い日時が一覧の主要カラムとして表示されない。
- 一覧から外した情報は、選択後に詳細ペイン、選択カード、tooltipのいずれかで確認できる。
- 詳細表示では対象レコードの全データを確認でき、問い合わせ・監査・障害調査に必要なID系情報が欠落しない。
- 日時表示は将来的に `YYYY/MM/DD HH:MM:SS` へ統一できる表示経路になっている。
- 検索・絞り込み対象はカラム削減で狭めない。
- RBAC、CRUD、DBスキーマ、public_id仕様は変更しない。

## 5. 共通テスト観点

- 各画面のテーブルヘッダーまたは候補表示が、本仕様の「一覧に残すカラム」に一致する。
- 削除対象カラムが一覧に表示されていない。
- 行選択後、削除対象カラムに含まれていた値を詳細ペイン、選択カード、tooltipで確認できる。
- 長い文字列、長い備考、長いID、長い日時を含むデータでも、一覧の主要カラムが読める。
- キーボード操作と行選択時の詳細更新が維持される。
- viewer/editor/admin の権限制御が既存仕様から変わらない。
- 既存の検索、登録、更新、削除、復元、完全削除、関連付け、監査ログ確認の代表UATが通る。
