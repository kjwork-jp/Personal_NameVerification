# 64_data_scale_and_asset_storage_policy.md

## 1. 目的

本書は、NameVerification v3 の v0.1.0 系における実データ件数規模の前提と、アイコン・画像などの外部資産保存方式を定義する。

対象:

- v0.1.0-rc2
- v0.1.0 系の後続パッチ
- v0.2.0 以降の性能・資産管理検討の初期前提

---

## 2. 結論

| 論点 | 決定 |
|---|---|
| v0.1.0 系の想定規模 | 小〜中規模ローカル利用 |
| 初期Go判定の必須性能試験 | 大量データ性能試験はblockerにしない |
| v0.1.0 系のアイコン保存 | DB保存しない |
| v0.1.0 系の画像/アイコン資産 | 実装対象外または外部ファイル参照の将来拡張扱い |
| v0.2.0 以降の検討 | assets配下の相対パス管理を第一候補とする |

---

## 5. アイコン・画像保存方式

### 5.1 v0.1.0 系の決定

v0.1.0 系では、アイコン・画像資産の保存は実装対象外とする。

理由:

- 現行の主要価値は名前・タイトル・サブタイトル・リンク・検索・監査・運用操作である。
- 画像資産をDBへ入れると、DBサイズ・backup/export・restoreの負荷が増える。
- portable ZIP配布では、画像ファイルをDB内に閉じ込めるより、将来assets配下の相対パス管理にした方が保守しやすい。

### 5.2 v0.2.0 以降の第一候補

将来アイコン・画像を扱う場合は、DBにバイナリ保存せず、portable package root配下の `assets/` に保存し、DBには相対パスを保持する。

```text
release/<ReleaseName>/
├─ assets/
│  ├─ icons/
│  └─ images/
└─ 30_prod_db/
   └─ nameverification.db
```

| 項目 | 方針 |
|---|---|
| 画像実体 | `assets/icons`, `assets/images` 配下 |
| DB保存内容 | 相対パス、表示名、mime/type、任意のhash |
| backup | DB backupとは別にassetsをコピー対象に含める |
| export | CSV/JSONには相対パスを出力 |
| restore | DB restoreとassets restoreの整合性確認が必要 |

---

## 6. 禁止・非推奨

| 内容 | 理由 |
|---|---|
| v0.1.0系で画像バイナリをSQLiteへ保存する | DB肥大化、backup/export負荷、UI劣化リスク |
| 絶対パスだけをDBに保存する | portable移動時にリンク切れになる |
| assets管理なしでUIだけ画像参照を追加する | 配布・backup・restore・exportの整合性が壊れる |

---

## 7. Go / No-Go 判断への影響

v0.1.0-rc2 のGo/No-Go判断では、以下はblockerにしない。

- 大規模データ性能試験
- アイコン/画像保存方式の実装
- assets管理機能

理由:

- v0.1.0 系の実運用範囲は小〜中規模ローカル利用である。
- アイコン/画像は現時点の主要業務導線に含めない。
- Day1 UATでは、現行機能の検索/CRUD/link/export/backup/restore/logを優先する。

## 9. 現時点の判定

v0.1.0-rc2では、実データ件数規模とアイコン保存方式は **未実装blockerではない**。

Day1 UATでは、実際に投入予定の最小データまたはサンプルデータで、検索・CRUD・export・backupが成立することを確認すればよい。
