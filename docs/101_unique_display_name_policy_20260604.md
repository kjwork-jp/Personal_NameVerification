# 101_unique_display_name_policy_20260604.md

## 1. 目的
2026-06-04 の UAT 画像レビューで、タイトル `test` の重複登録が確認された。
名前・タイトル・サブタイトルの表示名が重複すると、一覧やドロップダウンで ID を併記しない限り利用者が対象を識別できず、UI が複雑になる。

本書は、名前・タイトル・サブタイトルの重複登録禁止に向けた実装前仕様を定義する。
今回は docs-only とし、コード、DB、テストは変更しない。

## 2. 対象範囲
対象エンティティ:
- `names`
- `titles`
- `subtitles`

対象操作:
- 新規作成
- 更新
- 復元
- Import / restore

対象外:
- 既存コード変更
- DB migration 作成
- テスト追加・変更
- 重複検出スクリプト実装
- 既存データの実移行

## 3. 用語
| 用語 | 意味 |
| --- | --- |
| 表示名 | 利用者が入力し、一覧・検索・ドロップダウンに表示する値。例: `raw_name`, `title_name`, `subtitle_name` |
| 完全一致 | 表示名の文字列が完全に一致すること。前後空白、全角半角、大文字小文字の差分はそのまま差分として扱う |
| 正規化値 | 比較用に NFKC 正規化、制御文字除去、空白畳み込み、前後空白除去、大文字小文字同一視を適用した値 |
| 有効行 | `deleted_at IS NULL` の行 |
| 削除済み行 | `deleted_at IS NOT NULL` の行 |

## 4. 基本方針
- UI で ID を常時表示しなくても識別できる状態を保つため、利用者が主表示として読む名前・タイトル・サブタイトル名は重複登録させない。
- 重複判定の主キーは表示名そのものではなく、正規化値とする。
- 表示名完全一致は当然重複として拒否する。
- 表示名が完全一致しなくても、正規化値が一致する場合は重複として拒否する。
- 削除済み行は通常の新規作成・更新のユニーク判定に含めない。
- 復元時は、復元対象が有効行と衝突する場合に復元を拒否する。
- Import / restore は、取り込み結果が有効行のユニーク方針を満たさない場合に拒否する。
- 既存重複データがある DB では、DB 制約追加前に移行計画に沿って解消する。

## 5. 正規化ルール
比較用正規化値は、既存の `app/domain/normalization.py` の考え方に合わせる。

正規化処理:
- `None` は不正値として扱う。
- 制御文字は空白へ置換する。
- Unicode NFKC 正規化を行う。
- 連続する空白類は半角スペース 1 つへ畳み込む。
- 前後空白を除去する。
- 大文字小文字を同一視する。
- 正規化後に空文字になる値は登録不可とする。

例:
| 入力A | 入力B | 判定 |
| --- | --- | --- |
| `test` | `test` | 表示名完全一致のため重複 |
| `test` | ` TEST ` | 正規化値一致のため重複 |
| `ﾃｽﾄ` | `テスト` | 正規化値一致のため重複 |
| `A  B` | `a b` | 正規化値一致のため重複 |

## 6. 名前のユニーク方針
対象カラム:
- 表示名: `names.raw_name`
- 正規化値: `names.normalized_name`

方針:
- 有効行の `normalized_name` は全体で一意とする。
- 新規作成時、入力名の正規化値が有効な既存 `names.normalized_name` と一致する場合は拒否する。
- 更新時、更新後の正規化値が自分以外の有効行と一致する場合は拒否する。
- 復元時、復元対象の `normalized_name` が有効行と一致する場合は拒否する。
- 削除済み行同士の重複は、復元されるまでは許容する。

理由:
- 名前は検索・関連付けの起点であり、同じ正規化値の名前が複数あると照合結果の意味が曖昧になる。
- 既存 `docs/15_data_design.md` の「`normalized_name` は削除未済範囲で一意」という方針を維持する。

## 7. タイトルのユニーク方針
対象カラム:
- 表示名: `titles.title_name`
- 正規化値: 今後の実装で `titles.normalized_title_name` 相当を保持または式 index / service 計算で扱う

方針:
- 有効行のタイトル正規化値は全体で一意とする。
- 新規作成時、入力タイトル名の正規化値が有効な既存タイトルと一致する場合は拒否する。
- 更新時、更新後の正規化値が自分以外の有効タイトルと一致する場合は拒否する。
- 復元時、復元対象のタイトル正規化値が有効タイトルと一致する場合は拒否する。
- 削除済みタイトルは通常の新規作成・更新の重複判定に含めない。

理由:
- タイトルは一覧・タイトル選択・タイトル配下サブタイトル選択の親文脈として使われる。
- 同一タイトル名が複数あると、ID を出さない限り利用者が選択対象を判断できない。
- 2026-06-04 UAT で `test` の重複登録が確認されたため、実装で明示的に禁止する。

## 8. サブタイトルのユニーク方針
対象カラム:
- 親: `subtitles.title_id`
- 管理番号: `subtitles.subtitle_code`
- 表示名: `subtitles.subtitle_name`
- 正規化値: 今後の実装で `subtitles.normalized_subtitle_name` 相当を保持または式 index / service 計算で扱う

方針:
- 有効行の `subtitle_code` は、同一 `title_id` 配下で一意とする。
- 有効行のサブタイトル正規化値は、同一 `title_id` 配下で一意とする。
- サブタイトル名の重複禁止は全タイトル横断ではなく、親タイトル配下単位とする。
- 新規作成時、同一有効タイトル配下で入力サブタイトル名の正規化値が既存有効サブタイトルと一致する場合は拒否する。
- 更新時、同一有効タイトル配下で更新後の正規化値が自分以外の有効サブタイトルと一致する場合は拒否する。
- サブタイトルの親タイトル変更を許す実装では、移動先タイトル配下で `subtitle_code` とサブタイトル正規化値の両方を判定する。
- 復元時、復元対象の親タイトルが削除済みの場合は先にタイトル復元を要求する。
- 復元時、復元対象の `subtitle_code` またはサブタイトル正規化値が同一有効タイトル配下の有効サブタイトルと衝突する場合は拒否する。
- 削除済みサブタイトルは通常の新規作成・更新の重複判定に含めない。

理由:
- サブタイトルはタイトル配下の子要素であり、同じサブタイトル名が別タイトルに存在することは業務上あり得る。
- ただし同一タイトル配下で同じサブタイトル名が複数あると、一覧・関連付け・削除データで対象識別が難しくなる。
- 既存 `docs/15_data_design.md` の `title_id + subtitle_code` 一意方針に加えて、表示名側も同一タイトル配下で一意にする。

## 9. deleted_at の扱い
通常操作:
- `deleted_at IS NULL` の有効行のみをユニーク判定に含める。
- `deleted_at IS NOT NULL` の削除済み行は、新規作成・更新時のユニーク判定に含めない。

復元操作:
- 復元対象を `deleted_at IS NULL` に戻した結果、有効行のユニーク方針に違反する場合は復元を拒否する。
- 復元拒否時は、衝突している有効行の表示名と対象種別を UI に示す。

完全削除:
- 完全削除済みデータは DB 上に存在しないため、ユニーク判定対象外とする。

理由:
- 論理削除後に同じ表示名で再登録できないと、誤登録の作り直しが困難になる。
- 一方で復元時に衝突を許すと、有効行の一覧で同名が再発するため、復元時に止める。

## 10. 既存重複データの移行方針
DB 制約を追加する前に、既存 DB の重複を検出・解消する。

移行手順:
1. 本番相当 DB のバックアップを取得する。
2. 有効行のみを対象に、名前、タイトル、同一タイトル配下サブタイトルの正規化値重複を検出する。
3. 検出結果には、対象種別、ID、表示名、正規化値、`deleted_at`、関連数、最終更新日時を含める。
4. 業務判断で残す行を 1 件決める。
5. 残さない行は、表示名変更、関連付けの付け替え、論理削除、完全削除のいずれかで解消する。
6. 名前重複を解消する場合は、`name_subtitle_links` と `name_title_links` の関連先統合要否を確認する。
7. タイトル重複を解消する場合は、配下サブタイトルと `name_title_links` の移行先を確認する。
8. サブタイトル重複を解消する場合は、`name_subtitle_links` の移行先を確認する。
9. 重複が 0 件であることを再検出で確認する。
10. その後に DB 制約、service validation、UI 事前チェックを実装・有効化する。

移行中の禁止事項:
- 自動で片方を削除しない。
- ID の大小だけで残す行を決めない。
- 関連付けを確認せずに名前・タイトル・サブタイトルを統合しない。
- 重複が残ったまま DB 制約を追加しない。

## 11. UI 上のエラーメッセージ案
共通方針:
- `ConflictError` として扱う。
- 利用者が何を直せばよいか分かる文言にする。
- 可能な場合は衝突している既存表示名を示す。
- ID 常時表示に戻す前提の文言にしない。

名前:
- 新規作成: `同じ名前がすでに登録されています。別の名前に変更してください。`
- 更新: `更新後の名前は既存の名前と重複します。別の名前に変更してください。`
- 復元: `同じ名前の有効データがあるため復元できません。既存データを変更または削除してから再試行してください。`

タイトル:
- 新規作成: `同じタイトルがすでに登録されています。別のタイトル名に変更してください。`
- 更新: `更新後のタイトル名は既存のタイトルと重複します。別のタイトル名に変更してください。`
- 復元: `同じタイトル名の有効データがあるため復元できません。既存データを変更または削除してから再試行してください。`

サブタイトル:
- 新規作成: `選択中のタイトル内に同じサブタイトルがすでに登録されています。別のサブタイトル名に変更してください。`
- 更新: `更新後のサブタイトル名は選択中のタイトル内で重複します。別のサブタイトル名に変更してください。`
- 管理番号重複: `選択中のタイトル内に同じ管理番号がすでに登録されています。別の管理番号に変更してください。`
- 復元: `選択中のタイトル内に同じサブタイトルまたは管理番号の有効データがあるため復元できません。`

Import / restore:
- `取り込みデータに重複があります。名前・タイトル・サブタイトルの重複を解消してから再実行してください。`
- `既存データと取り込みデータが重複します。重複一覧を確認してから再実行してください。`

## 12. DB 制約 / service validation / UI 事前チェックの役割分担
| 層 | 役割 | 必須度 | 補足 |
| --- | --- | --- | --- |
| DB 制約 | 最終防衛線として有効行の重複を書き込ませない | 必須 | 同時操作、Import、将来の別 UI 経由更新を防ぐ |
| service validation | 業務ルールを一元化し、利用者向けエラーに変換する | 必須 | `ConflictError` として UI が扱える粒度にする |
| UI 事前チェック | 入力直後または保存前に早く気づかせる | 推奨 | UX 改善目的。DB / service の代替にはしない |

DB 制約の実装方針:
- SQLite では有効行のみを対象にした partial unique index を第一候補とする。
- 名前: `normalized_name` の `deleted_at IS NULL` partial unique index。
- タイトル: タイトル正規化値の `deleted_at IS NULL` partial unique index。
- サブタイトル管理番号: `(title_id, subtitle_code)` の `deleted_at IS NULL` partial unique index。
- サブタイトル名: `(title_id, サブタイトル正規化値)` の `deleted_at IS NULL` partial unique index。
- 正規化値カラムを追加するか、式 index で扱うかは migration 実装時に決定する。ただし service と DB で正規化ルールがずれないことを必須とする。

service validation の実装方針:
- create / update / restore / import / restore backup のすべてで同じ判定関数を使う。
- DB 制約違反をそのまま UI へ出さず、対象種別別の `ConflictError` に変換する。
- 更新時は自分自身の行を重複判定から除外する。
- 削除済み行は通常操作の重複判定から除外する。
- 復元時は復元後の有効状態を前提に判定する。

UI 事前チェックの実装方針:
- 保存ボタン押下前に、入力値の正規化結果が既存有効行と衝突する可能性を示す。
- 事前チェック結果は補助表示とし、最終判定は service / DB に委ねる。
- 一覧やドロップダウンで ID 併記を必須にする方向ではなく、重複を作らせない方向で UI を単純化する。

## 14. レビュー観点
- サブタイトル名の一意範囲が「全体」ではなく「同一タイトル配下」でよいか。
- 削除済み行を通常操作のユニーク判定から除外する方針でよいか。
- タイトル正規化値、サブタイトル正規化値を永続カラムで持つか、式 index / service 計算で扱うか。
- Import / restore 時に全件拒否とするか、重複行だけをスキップする運用を許すか。
- 既存重複データ解消時に、関連付け統合の業務判断者を誰にするか。
