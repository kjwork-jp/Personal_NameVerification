# UI/UX 再設計方針

## 目的

NameVerification v3 を、開発者向けのDB管理ツールではなく、EXE配布を前提にした業務アプリとして使える状態に近づける。

## 現状課題

- 内部ID、コード、relation_type、JSONなどの内部実装用語が画面に出すぎている。
- タイトルとサブタイトルが同一画面にあり、操作順が初見で分かりにくい。
- リンク管理、監査ログ、運用操作の目的が画面上で説明されていない。
- 全体が暗く、表・入力欄・危険操作・通常操作の区別が弱い。
- 操作者IDなど、将来的には設定・ログイン・EXE起動時情報から補完すべき項目が手入力になっている。

## Phase 1 の対象

この段階では大きなDB変更や画面全面刷新は行わない。既存の通過済み品質ゲートを壊さない範囲で、以下を実施する。

- 共通テーマを追加し、画面の余白・ボタン・タブを見やすくする。
- タブ名を業務向け日本語に変更する。
- 検索画面の内部ID露出を減らす。
- 「紐づき」「関連行」「監査ログ」「リンク管理」などの文言を、利用者向けの説明に置換する。
- 関連付け画面と操作履歴画面に説明ヘッダーを追加する。

## Phase 2 の対象

タイトル管理とサブタイトル管理を、画面上の導線として分離する。

- タイトル管理: 最初に名前を選び、その名前に関連するタイトルを登録・更新する。
- サブタイトル管理: 最初にタイトルを選び、そのタイトル配下のサブタイトルを登録・更新する。
- 既存の `TitleSubtitleManagementTab` は互換性維持のため内部エディタとして残す。
- 画面上は `TitleManagementTab` と `SubtitleManagementTab` の2タブへ分ける。
- DB変更やUUID移行はこのフェーズでは行わない。

## Phase 3 以降の方針

### 1. ID / コード入力廃止

- 画面からID・コードの手入力欄を原則撤去する。
- 内部IDはUUIDへ寄せる。
- 既存の整数IDを即時削除するのではなく、移行フェーズでは内部互換を維持する。
- サブタイトルコードはユーザー入力ではなく、自動生成または内部値として扱う。

### 2. 一意制約

- 名前は一意。
- タイトル名は一意。
- サブタイトル名は原則としてタイトル配下で一意。
- 全体検索性を優先する場合は、サブタイトル名の全体一意化も検討する。

### 3. 関連付け管理

- 通常登録ではなく、例外的な関連修正画面として扱う。
- 「リンク」ではなく「関連付け」に統一する。
- `relation_type` は画面上では「関連の種類」と表示する。

### 4. 操作履歴

- 「監査ログ」ではなく「操作履歴」として表示する。
- JSONは通常表示ではなく、詳細確認用として扱う。
- 「誰が、いつ、何をしたか」を主表示にする。

### 5. EXE配布前提

- DB保存場所を明示する。
- 初回起動時のDB作成を案内する。
- バックアップ先を既定化する。
- ログ保存場所を画面またはヘルプに表示する。
- ヘルプ画面、バージョン表示、設定画面を追加する。

## 推奨実装順

1. Phase 1: UIテーマ、文言、説明ヘッダー、ID露出削減。
2. Phase 2: タイトル管理とサブタイトル管理の画面分離。
3. Phase 3: ID/コード手入力廃止、UUID内部化の設計・移行。
4. Phase 4: 関連付け管理と操作履歴の高度化。
5. Phase 5: EXE化、保存先、初回起動、ヘルプ、マニュアル更新。

## 完了条件

- `pytest -q` が通る。
- `ruff check .` が通る。
- `black --check .` が通る。
- `mypy app` が通る。
- 初見で、各画面の目的が画面上の説明から分かる。
- IDや内部コードを利用者が意識しない導線になっている。
