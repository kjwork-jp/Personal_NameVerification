# CONTRIBUTING.md

## 目的
本ドキュメントは、NameVerification v3 リポジトリにおける変更手順、レビュー手順、AI を含む開発分担手順を定義します。

## 前提
- コード実装は設計文書の承認後に開始する
- 仕様変更と実装変更は同一コミットに混在させない
- ドキュメント更新が必要な変更は、コード変更より先に docs を更新する

## 変更フロー
1. 対象タスクを `docs/30_development_process_map.md` で確認
2. `docs/02_document_priority_order.md` に従い、参照文書を読み込む
3. `docs/35_definition_of_ready.md` を満たすか確認する
4. ブランチを作成する
5. 小さな単位で変更する
6. ローカルテストを実施する
7. `docs/43_review_checklist.md` でセルフレビューする
8. PR を作成する
9. PR 本文に変更範囲・根拠・テスト結果・残課題を書く
10. マージ前に `docs/44_final_review_checklist.md` を確認する

## 役割分担
- ChatGPT:
  - 要件定義
  - 設計
  - 総合レビュー
  - 運用開始前チェック
- Codex:
  - 実装
  - 単体テスト
  - 初期結合テスト
  - PR 作成
- Human owner (kjwork-jp):
  - 最終承認
  - マージ判断
  - 運用開始判定

## 禁止事項
- main へ直接 push
- 文書未読での実装着手
- 仕様未確定のまま UI/DB を先行確定
- 不要なライブラリ追加
- 1 PR に複数目的を混在
- テスト未実施のまま PR 作成
- 「PoC」「temporary」「workaround」を恒久実装に残すこと

## 推奨事項
- 変更前後の差分を説明できる状態にする
- 仕様根拠は docs のパスで示す
- 命名・責務・例外処理を先に整える
- リファクタは機能変更と分離する

## PR 提出前の最低確認
- [ ] 変更目的が 1 つに絞られている
- [ ] 対応する docs が更新済み
- [ ] テスト結果が記載されている
- [ ] リスクと未実施項目が明記されている
- [ ] セルフレビュー済み
